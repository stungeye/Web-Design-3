# Structure Content With Semantic HTML

Web Design 3 is about building small professional websites that are planned clearly, built semantically, styled intentionally, tested, and published.

<Note>
  Good layout starts before CSS. If the content has no clear structure, the
  visual design has to fight the markup instead of styling it.
</Note>

![Simple page regions diagram](./images/page-regions.svg)

## The WD3 Build Process

Each assignment follows the same practical workflow:

```text
1. Structure the content.
2. Style readable defaults.
3. Create intentional layouts.
4. Plan responsive behaviour.
5. Build responsive sections.
6. Organize the CSS.
7. Test the site.
8. Publish a working static version.
```

<Practice>Treat planning as part of the build.</Practice>

## What WD3 Covers

WD3 focuses on:

- semantic HTML
- accessible document structure
- readable base CSS
- practical Grid and Flexbox layout
- fluid responsive design
- basic testing
- simple static deployment

Not the focus: deep SEO, analytics, production hosting infrastructure, advanced design systems, or JavaScript application development.

## Layout Starts as Reading Order

Good layout starts as reading order, grouping, and hierarchy.

Before choosing Grid, Flexbox, spacing, or colours, decide:

- what the page is about
- what belongs together
- what order the content should be read in
- which parts are global site structure
- which parts are unique page content

## Structure Before Style

HTML describes the meaning and organization of content. CSS describes presentation.

```html
<article>
  <h2>Guided Walking Tours</h2>
  <p>Join a local guide for a two-hour tour.</p>
  <a href="tours.html">View walking tour times</a>
</article>
```

<Warning>
  Do not choose headings, lists, or buttons because of how they look by default.
  Choose them because they match the content's job.
</Warning>

## Diagnostic Questions

Before writing markup, ask:

- What is the page about?
- What content is global to the site?
- What content is unique to this page?
- What is the main content?
- Which sections need headings?
- Which links must be clear out of context?
- Which images communicate information?

<Practice>
  Remove all CSS and ask whether the page still reads in a logical order.
</Practice>

## Element Choices Should Match Content

The previous example works because each element has a clear job:

- `<article>` groups a self-contained content item.
- `<h2>` introduces the item.
- `<p>` contains paragraph text.
- `<a>` goes somewhere and says where it goes.

Classes are still important, but they should not replace meaningful elements. A class like `class="button"` can style a link, but it cannot turn non-link text into a real link.

<Practice>
  Use native elements first. Add classes when styling needs them.
</Practice>

## Page Anatomy

Most small brochure pages have a familiar structure:

```text
body
  header
    site name
    navigation
  main
    h1
    content sections
  footer
```

<Note>
A page should usually have one primary `<main>` element. The `main` landmark contains the content that is unique to that page.
</Note>

## Page Anatomy In HTML

```html
<body>
  <header>
    <a href="index.html">Assiniboine Park</a>
    <nav aria-label="Main navigation">...</nav>
  </header>

  <main>
    <h1>Plan Your Visit To Assiniboine Park</h1>
    <section>
      <h2>What To See</h2>
      <p>Explore gardens, trails, and events.</p>
    </section>
  </main>

  <footer>...</footer>
</body>
```

## Structure Creates Layout Handles

Semantic structure gives CSS meaningful regions to style later.

```html
<main>
  <section class="hero">...</section>
  <section class="card-grid">...</section>
  <section class="visit-info">...</section>
</main>
```

The classes help with presentation, but the structure says what the content is and how the page should be understood.

## Landmarks

Landmarks identify major page regions:

- `<header>` for introductory site or section content
- `<nav>` for major navigation links
- `<main>` for the page's primary content
- `<aside>` for related or complementary content
- `<footer>` for closing site or section information

<Warning>
  More landmarks do not automatically mean better accessibility. Use landmarks
  when they identify real regions.
</Warning>

## Headings Create The Outline

Headings describe the page outline. A user should be able to skim the headings and understand what the page contains.

```html
<h1>Plan Your Visit To Assiniboine Park</h1>

<section>
  <h2>Hours And Admission</h2>
  <p>The park is open daily.</p>
</section>
```

<Practice>
  Write headings as labels for real content sections, not as decoration.
</Practice>

## Nested Headings

Use nested headings when content belongs inside a larger section:

```html
<section>
  <h2>Featured Attractions</h2>
  <article>
    <h3>The Leaf</h3>
    <p>Visit indoor biomes, gardens, and exhibits.</p>
  </article>
  <article>
    <h3>Leo Mol Sculpture Garden</h3>
    <p>Walk through sculptures and landscaped paths.</p>
  </article>
</section>
```

Headings should not skip levels for visual effect.

## Sectioning Content

Use sectioning elements when they clarify the content model:

- `<section>` for a thematic part of a page that needs a heading
- `<article>` for content that could stand alone or be reused
- `<aside>` for related content that supports nearby content
- `<div>` when you only need a styling hook

It is easy to overuse `<section>` because it sounds more meaningful than `<div>`. If you cannot write a useful heading for the group, it may not need to be a section.

## Rewrite Messy Markup

The week 1 activity is to rewrite messy brochure content as semantic HTML with no CSS.

The goal is not visual polish. The goal is a page that reads logically when unstyled.

```html
<div class="top">
  <div class="big">Visit Gimli</div>
  <div><span>Home</span> <span>Events</span></div>
</div>
<div class="section-title">Summer Festivals</div>
<div>Enjoy music, food, markets, and lakefront events.</div>
<div class="button">Click here</div>
```

## Semantic Rewrite

Ask what each piece of content is:

```html
<header>
  <a href="index.html">Visit Gimli</a>
  <nav aria-label="Main navigation">
    <ul>
      <li><a href="index.html">Home</a></li>
      <li><a href="events.html">Events</a></li>
    </ul>
  </nav>
</header>

<main>
  <h1>Visit Gimli</h1>
  <section>
    <h2>Summer Festivals</h2>
    <p>Enjoy music, food, markets, and lakefront events.</p>
    <a href="events.html">Explore summer festivals</a>
  </section>
</main>
```

## A Useful Semantic Plan

A semantic plan can be a nested list or simplified HTML skeleton.

```text
Page 1: Main brochure page
header
  site name linking home
  main navigation
main
  h1: Visit Gimli, Manitoba
  section: Why visit Gimli
  section: Lakefront activities
  section: Festivals and events
  section: Local food and shops
footer
  source credits
  secondary links
```

<Practice>
  Before moving to Figma or CSS, read the structure plan from top to bottom.
</Practice>
