# Sliders Architecture Onboarding

Sliders is a static teaching system that turns one authored MDX module into two outputs: student notes and classroom slides. The project is intentionally small, but the implementation has a few important boundaries that make it predictable to extend.

<Note>
Read this module as a map of the codebase, not just a tour of files. The key idea is that content, rendering, validation, demos, and deployment each have a narrow job.
</Note>

## The Product Shape

The application publishes a course index, a notes page for every module, a slide deck for every module, and standalone full-screen demo pages.

```text
/                                      course index
/02-css-layout/01-grid/                notes view
/02-css-layout/01-grid/slides/         slide view
/demos/GridExplorer/                   standalone demo view
```

The same MDX file feeds both the notes and slides views. Authors do not maintain a separate slide deck.

<Aside>
This is the main product decision. Sliders is not a general presentation builder. It is a static course-content pipeline. The system assumes that the notes and slides are two views over one module source, so implementation work should protect that invariant.
</Aside>

<Practice>
When changing behavior, ask whether the change belongs to content discovery, source parsing, MDX component rendering, client-side slide behavior, or demo enhancement.
</Practice>

## The Triple Debt Lens

This module borrows from Margaret-Anne Storey's triple debt model: software health depends on code quality, shared understanding, and captured intent. In Sliders, all three matter because the project is designed to be changed by humans and AI-assisted agents.

```text
technical debt   code is harder to change
cognitive debt   the system is harder to understand
intent debt      the system's purpose and constraints are unclear
```

For this project, technical debt shows up as brittle parsing, duplicated route behavior, fragile browser DOM manipulation, untested demo state, or CSS changes that improve one view while breaking another.

Cognitive debt shows up when a maintainer can name the files but cannot explain the flow from MDX source to notes, slides, validation, and demo enhancement.

Intent debt shows up when someone can make a technically correct change but cannot tell whether it serves the product: one authored module, two useful outputs, static hosting, explicit demos, accessible course material, and low-friction authoring.

<Aside>
The relevant paper is "From Technical Debt to Cognitive and Intent Debt: Rethinking Software Health in the Age of AI" by Margaret-Anne Storey: [arXiv:2603.22106](https://arxiv.org/pdf/2603.22106). The useful local takeaway is that passing tests is not enough. A future maintainer also needs the mental model and the decision record that make those tests meaningful.
</Aside>

<Warning>
AI can reduce code-level effort while increasing understanding and intent risk. Treat explanations, tests, and rationale as part of the deliverable, not decoration after the implementation is done.
</Warning>

## Astro And Route Architecture

Astro owns routing, static generation, layouts, and build output. The project uses file-based routes under `src/pages`.

```text
src/pages/index.astro
src/pages/[unit]/[module]/index.astro
src/pages/[unit]/[module]/slides/index.astro
src/pages/demos/[demo]/index.astro
```

The bracketed folders are dynamic route parameters. During the static build, each route exports `getStaticPaths`, asks the content discovery layer for known modules, and returns one route per module.

<Aside>
If you come from React apps, think of Astro pages as server-rendered route modules that run at build time. The frontmatter block at the top of an `.astro` file is where data is loaded and components are imported. The HTML-like template below that block is the rendered page.
</Aside>

```astro
---
const courseIndex = await getCourseIndex();

export async function getStaticPaths() {
  return courseIndex.units.flatMap((unit) =>
    unit.modules.map((module) => ({
      params: { unit: unit.slug, module: module.slug },
      props: { module },
    })),
  );
}
---
```

<Warning>
The URL base is configured for GitHub Pages. Links that leave route files should pass through `toSitePath` so local development and the `/Sliders` production base path stay aligned.
</Warning>

## Content Is The Source Of Truth

Course modules live directly under `src/content/units`. Each unit is a folder, and each direct `.mdx` file is a module.

```text
src/content/units/
  00-project-onboarding/
    01-sliders-architecture.mdx
  01-semantic-html/
    01-introduction.mdx
    images/
      page-regions.svg
  02-css-layout/
    01-grid.mdx
```

Numeric prefixes control ordering and remain in URLs. Visible module titles come from the module's first H1. Visible unit titles are inferred from folder slugs by `formatDisplayTitle`.

<Aside>
Subfolders inside a unit are currently asset folders, not nested module routes. The authoring validator rejects nested `.mdx` files because the router and discovery code only support direct module files.
</Aside>

The supported module structure is convention-based:

```mdx
# Module Title

Opening notes become part of the title slide.

## First Slide And Notes Section

This starts a slide and a major notes section.

### Nested Heading

This stays inside the current slide or notes section.
```

## Discovery And Parsing

Two layers read module files for different reasons.

`contentDiscovery.js` finds unit folders and direct module files. It builds the course index data used by the home page and dynamic route generation.

`moduleParser.js` reads a module's source text and turns H1 and H2 headings into structured slide metadata.

```text
MDX file
  -> contentDiscovery.js finds it and reads the H1 title
  -> moduleNotes.js validates it and builds table of contents data
  -> moduleSlides.js validates it and counts slides
  -> Astro imports the MDX component and renders the actual page content
```

<Aside>
The parser does not render MDX. It only understands enough Markdown structure to find top-level headings outside fenced code blocks. Rendering remains Astro and MDX's job. This keeps the parser testable and avoids building a partial Markdown renderer.
</Aside>

<Note>
The title slide is made from the H1 plus any opening content before the first H2. Every H2 after that becomes one slide and one notes table-of-contents entry.
</Note>

<Practice>
When changing the parser, write the test as a statement of the authoring contract. A parser test should explain which source shape is supported, rejected, or intentionally ignored.
</Practice>

## Notes Rendering Pipeline

The notes route renders the original MDX as a readable article with a generated table of contents.

```text
src/pages/[unit]/[module]/index.astro
  -> getCourseIndex()
  -> import.meta.glob() lookup for the MDX component
  -> getModuleNotesData()
  -> createNotesMdxComponents()
  -> BaseLayout
```

`getModuleNotesData` validates demo references, parses the module, and builds section IDs for the table of contents. The route then renders the imported MDX component inside the notes layout.

<Aside>
There are two parallel representations in play. The parsed representation supplies metadata such as title, sections, and table of contents. The imported MDX component supplies the actual rendered HTML and React components. This split is deliberate because Astro already knows how to render MDX correctly.
</Aside>

In notes mode, H2 elements receive stable IDs, teaching primitives render as callouts, and demo components are wrapped in a figure with a full-screen link.

<Practice>
If a notes-only feature depends on section structure, add or adjust it near `getModuleNotesData`, `headingIds.js`, or `createNotesMdxComponents.jsx`.
</Practice>

## Slide Rendering Pipeline

The slide route initially renders the same MDX content into a hidden source container. A small browser controller then groups the rendered H1 and H2 regions into slide sections.

```text
src/pages/[unit]/[module]/slides/index.astro
  -> getCourseIndex()
  -> import.meta.glob() lookup for the MDX component
  -> getModuleSlidesData()
  -> createSlidesMdxComponents()
  -> setupSlideDeck(document)
```

The slide controller runs in the browser because it reorganizes rendered DOM nodes. It finds H1 and H2 elements, moves each heading and the following content into a slide section, hides inactive slides, updates status text, and handles keyboard navigation.

```text
Right, Down, PageDown, Space     next slide
Left, Up, PageUp, Shift+Space    previous slide
Home                             first slide
End                              last slide
```

<Warning>
The expected slide count comes from source parsing, while the actual slide sections come from rendered DOM. `assertExpectedSlideCount` catches drift between those two views of the module.
</Warning>

<Aside>
This design means the slide deck does not need to compile every section separately. The tradeoff is that the browser owns the final grouping step. Tests around `slideDeckController.js` are important because a small DOM change can affect all slide decks.
</Aside>

## MDX Components And Teaching Primitives

The MDX component layer decides how author-facing tags render in each view.

```text
src/components/mdx/MdxComponents.jsx
src/components/mdx/createNotesMdxComponents.jsx
src/components/mdx/createSlidesMdxComponents.jsx
```

The shared teaching primitives are Note, Warning, Question, Practice, and Wait. They render as labeled callouts in both notes and slides.

Aside is intentionally different. It appears in notes and returns nothing in slides.

```jsx
export const notesMdxComponents = {
  ...sharedMdxComponents,
  Aside,
};

export const slidesMdxComponents = {
  ...sharedMdxComponents,
  Aside: HiddenAside,
};
```

<Aside>
This is the cleanest place to make view-specific authoring behavior. If instructors need presenter-only content or student-only content later, model it here first before adding more parser rules.
</Aside>

<Note>
The authoring validator treats uppercase MDX tags as either built-in teaching primitives or registered demos. Unknown component tags fail validation before publishing.
</Note>

## React Demos And Client Enhancement

Interactive demos are regular React components registered explicitly in the demo registry.

```text
src/demos/GridExplorer/
  GridExplorer.jsx
  GridExplorer.css
  gridExplorerModel.js
  gridExplorerClient.js

src/lib/demoMetadata.js
src/lib/demoRegistry.js
src/lib/demoClientEntrypoint.js
```

The React component provides deterministic initial markup. Optional browser behavior is attached later by a client enhancer. The shared client entrypoint imports an enhancer only if the page contains a matching demo root.

```js
const demoEnhancers = [
  {
    selector: "[data-grid-explorer]",
    load: async () => {
      const module = await import("../demos/GridExplorer/gridExplorerClient.js");
      return module.setupGridExplorerDemos;
    },
  },
];
```

<Practice>
When adding a demo, add metadata, register the component, add any client enhancer, and cover the model or interaction behavior with focused tests.
</Practice>

<Aside>
The project avoids letting arbitrary local MDX imports define demos. That keeps authoring simple, makes unknown component references detectable, and gives the standalone demo route a single registry to read from.
</Aside>

## Intent And Product Constraints

Sliders exists to help instructors author course material once and publish it as both long-form notes and lecture slides. That purpose should guide implementation choices more than framework novelty or feature breadth.

The durable constraints are:

```text
one direct MDX file per module
H1 creates the title slide
H2 creates a slide and notes section
notes and slides render from the same source
demos are registered explicitly
the site builds to static files
internal links respect the configured base path
accessibility is part of the output contract
```

The important non-goals are also part of the intent:

```text
not a general LMS
not a drag-and-drop slide editor
not a runtime-backed application
not a system where every module imports custom local components
not a parser for arbitrary Markdown and MDX structure
```

<Aside>
These constraints are intent artifacts. If one changes, update this module, the README, and the tests that encode the old assumption. Otherwise future maintainers and agents will preserve obsolete constraints or violate current ones without noticing.
</Aside>

<Practice>
When proposing a feature, write the sentence "This supports one-source teaching modules by..." before choosing the implementation.
</Practice>

## Validation And Tests

There are two safety nets: authoring validation for content conventions and Vitest tests for implementation behavior.

```text
npm run validate:authoring
npm test
npm run build
```

Authoring validation checks module shape, direct file placement, plain top-level headings, and unresolved uppercase MDX component tags.

Implementation tests cover parsing, content discovery, heading IDs, URL base behavior, demo registration, live code behavior, demo models, and slide controller behavior.

<Aside>
For code changes, prefer adding the smallest test that proves the behavior boundary. Parser tests should stay close to `moduleParser.test.js`; routing and discovery behavior belongs near `contentDiscovery.test.js` or `siteUrls.test.js`; browser DOM behavior belongs in jsdom-backed component or controller tests.
</Aside>

<Warning>
`npm run build` runs `astro check` before `astro build`, so TypeScript and Astro integration problems can appear there even when unit tests pass.
</Warning>

## Styling And Layout

Global CSS lives in `src/styles/global.css` and is loaded by `BaseLayout`. The project currently favors a single global stylesheet over component-scoped Astro styles.

```text
BaseLayout.astro
  -> global.css
  -> optional demo client script
  -> page body
```

The major styling surfaces are course index, notes layout, callouts, notes demo frames, slide shell, live code blocks, and demo-specific CSS.

<Aside>
Because notes and slides render the same MDX, CSS should be scoped by view-level classes when behavior diverges. For example, slide styling hangs from the slide page body class and slide-specific containers, while notes styling hangs from notes layout classes.
</Aside>

<Practice>
Before changing typography or content spacing, test both notes and slides. A style that improves long-form notes can easily make slide content overflow.
</Practice>

## Deployment Model

The deployed site is fully static. Astro builds the project into `dist`, and GitHub Actions publishes it to GitHub Pages.

```text
source push to main
  -> .github/workflows/deploy.yml
  -> npm ci
  -> npm run build
  -> publish dist to GitHub Pages
```

Production URLs use the `/Sliders` base path from `astro.config.mjs`.

<Note>
If the repository name or hosting target changes, check `site`, `base`, and every place route links are generated. The project already centralizes internal URL construction in `siteUrls.js`.
</Note>

## Debt Checks Before A Change

Use this checklist before and after non-trivial changes. It is meant to keep code, understanding, and intent aligned.

```text
technical
  What behavior boundary am I changing?
  Which focused test proves it?
  Could the change break both notes and slides?
  Could it break GitHub Pages base-path output?

cognitive
  Can I trace the data flow in plain language?
  Did I leave names and tests that explain the model?
  Would a new maintainer know where this behavior lives?
  Did I avoid hiding important behavior in clever glue code?

intent
  Which product goal does this serve?
  Which authoring constraint does it preserve or change?
  Is the rationale recorded in README, docs, tests, or this module?
  Would an AI agent have enough context to continue safely?
```

<Aside>
The strongest debt reduction artifact is often a small test with a precise name plus a short note explaining why the behavior exists. Tests preserve executable intent; notes preserve rationale; walkthrough-friendly code preserves shared understanding.
</Aside>

<Warning>
Do not let generated explanations stand in for understanding. Use AI to draft, compare, and check, but still trace the code path yourself before accepting a design.
</Warning>

## Taking Over Development

Start with the content contract, then follow the data flow from index discovery to route rendering to client enhancement.

```text
1. Read README.md and this module.
2. Skim the implementation spec and phase docs for original decisions.
3. Run npm test and npm run validate:authoring.
4. Open the course index in dev mode.
5. Compare one notes page with its slide page.
6. Trace the module through contentDiscovery, moduleNotes, moduleSlides, and the Astro route.
7. Inspect one demo from registry to React component to client enhancer.
8. Explain the system back from memory, then check what you missed.
```

The most important files to know first are:

```text
src/lib/contentDiscovery.js
src/lib/moduleParser.js
src/lib/moduleNotes.js
src/lib/moduleSlides.js
src/pages/[unit]/[module]/index.astro
src/pages/[unit]/[module]/slides/index.astro
src/components/mdx/MdxComponents.jsx
src/lib/demoRegistry.js
src/lib/slideDeckController.js
src/styles/global.css
```

<Practice>
For most feature work, keep the source MDX contract stable unless the product explicitly needs a new authoring primitive. A small rendering change is usually safer than expanding what the parser accepts.
</Practice>

<Aside>
If a change uncovers a hidden assumption, externalize it. Put stable product intent in README or this onboarding module, implementation rationale in docs or tests, and executable expectations in Vitest or authoring validation.
</Aside>

<Wait>
The architecture is simple on purpose. The hard part is preserving the guarantee that one human-authored module remains enough to produce useful notes, useful slides, and validated static output.
</Wait>
